# V6.5.6 Sierra Bracket Verification

## What was fixed

Backend `/trade/execute` was OVERWRITING the frontend's t1/t2/t3 targets
with recalculated values using T1_RR=1.5 and T1_MIN_PT=10.0. This caused:
- SHORT @ 7157.50 with C1=7153.75 got overwritten to t1=7147.50 (entry - 10pt)
- Sierra received wrong target, placed bracket at unexpected price

Fix: backend now forwards frontend's t1/t2/t3 directly. Only recomputes
if targets are missing/zero (manual trade fallback).

## Verification Steps

### Step 1: Start Bridge
```bash
cd /Users/michael/Downloads/mems26_web_git/bridge
export MEMS26_ENTRY_MODE=DEMO
python3 json_bridge.py 2>&1 | tee /tmp/bridge.log
```

### Step 2: Execute a test trade via Dashboard
1. Open Dashboard, navigate to a setup
2. Click EXECUTE on Pre-Entry Checklist
3. Note the Entry, Stop, C1, C2, C3 values shown in the UI

### Step 3: Check trade_command.json
```bash
cat /Users/michael/SierraChart2/Data/trade_command.json | python3 -m json.tool
```

Verify:
- `price` matches the Entry value from UI
- `t1` matches C1 from UI (NOT entry - 10pt)
- `t2` matches C2 from UI
- `t3` matches C3 from UI
- `qty` = 3
- `cmd` = "BUY" or "SELL"

### Step 4: Check Sierra Chart Trade Window
After DLL processes the command:
- THREE separate pending/filled orders should appear
- Each order: qty=1
- Each order: same stop price
- Order 1: target = C1 value
- Order 2: target = C2 value
- Order 3: target = C3 value

### Troubleshooting
If trade_command.json doesn't appear:
- Check Bridge log: `tail -f /tmp/bridge.log | grep C4`
- Check Redis: command should be in `mems26:trade:command`
- Check Render is awake: `curl https://mems26-web.onrender.com/health`

If targets in trade_command.json don't match UI:
- The backend fix may not have deployed yet
- Wait 3 min for Render auto-deploy, retry
